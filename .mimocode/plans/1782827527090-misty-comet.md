# Keycloak Integration Plan

## Overview

Replace the custom JWT authentication system with Keycloak OIDC integration. The frontend authenticates directly with Keycloak (public client), and the backend acts as a Resource Server validating Keycloak-issued JWT tokens.

**Keycloak Details:**
- Server: `http://172.29.11.201:8086`
- Realm: `Invoicing System`
- Client ID: `invoicingsystemclient`
- Client type: Public

## Phase 1: Backend тАФ Resource Server

### 1.1 Add dependencies to `pom.xml`

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-oauth2-resource-server</artifactId>
</dependency>
```

Remove the old `jjwt-*` dependencies (lines 68-84) тАФ no longer needed.

### 1.2 Update `application.properties`

Add Keycloak configuration:

```properties
spring.security.oauth2.resourceserver.jwt.issuer-uri=http://172.29.11.201:8086/realms/Invoicing%20System
spring.security.oauth2.resourceserver.jwt.jwk-set-uri=http://172.29.11.201:8086/realms/Invoicing%20System/protocol/openid-connect/certs
```

Remove old JWT properties (`jwt.secret`, `jwt.expiration`).

### 1.3 Rewrite `SecurityConfig.java`

Replace the entire file with OAuth2 Resource Server configuration:

- Remove `UserDetailsServiceImpl`, `AuthenticationFilter`, `AuthEntryPoint` dependencies
- Use `oauth2ResourceServer` with JWT configuration
- Map Keycloak realm roles (`ADMIN`, `USER`) to Spring Security authorities with `ROLE_` prefix
- Keep CORS config, CSRF disabled, stateless sessions
- Keep public endpoints: `/login` can be removed or repurposed, `/stop`, static assets, Swagger

### 1.4 Delete files no longer needed

- `service/JwtUtil.java` тАФ replaced by Spring's JWT decoder
- `config/JwtProperties.java` тАФ no longer needed
- `config/AuthenticationFilter.java` тАФ replaced by Resource Server filter
- `config/AuthEntryPoint.java` тАФ Spring handles 401s automatically
- `controller/LoginController.java` тАФ Keycloak handles login
- `service/UserDetailsServiceImpl.java` тАФ no local user lookup
- `record/AccountCredentialRecord.java` тАФ no local login endpoint
- `repository/UserRepository.java` тАФ no local user lookup (keep `UserEntity`, `RoleEntity` if used elsewhere for app data)

### 1.5 Role mapping

Keycloak JWT claims will contain roles in `realm_access.roles`. Configure Spring to extract them:

```java
jwt.jwtAuthenticationConverter(jwt -> {
    Collection<String> roles = jwt.getClaimAsStringList("realm_access.roles");
    List<SimpleGrantedAuthority> authorities = roles.stream()
        .map(role -> new SimpleGrantedAuthority("ROLE_" + role))
        .collect(Collectors.toList());
    return new UsernamePasswordAuthenticationToken(jwt.getSubject(), null, authorities);
});
```

The existing `@PreAuthorize("hasAnyRole('ADMIN', 'USER')")` annotations will work unchanged.

## Phase 2: Frontend тАФ OIDC Authentication

### 2.1 Install `oidc-client-ts`

```bash
cd "Backend/Invoicing System/src/main/Frontend Codebase"
npm install oidc-client-ts
```

### 2.2 Create `src/base/KeycloakService.ts`

New file that wraps `oidc-client-ts`:

- Configure with Keycloak realm endpoints
- `signinRedirect()` тАФ redirect to Keycloak login
- `signinRedirectCallback()` тАФ handle callback, store tokens
- `getUser()` тАФ get current user info
- `getToken()` тАФ get access token
- `signoutRedirect()` тАФ logout from Keycloak
- Handle token refresh automatically via `UserManager`

Configuration:
```typescript
const oidcConfig = {
  authority: "http://172.29.11.201:8086/realms/Invoicing%20System",
  client_id: "invoicingsystemclient",
  redirect_uri: window.location.origin + "/login",
  post_logout_redirect_uri: window.location.origin + "/login",
  response_type: "code",
  scope: "openid profile email"
};
```

### 2.3 Update `src/base/Api.ts`

Replace `sessionStorage` token retrieval with Keycloak token:

```typescript
import { getKeycloakService } from './KeycloakService';

api.interceptors.request.use(async config => {
  const token = await getKeycloakService().getToken();
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});
```

### 2.4 Update `src/base/AuthService.ts`

Replace `loginApi`, `isTokenExpired`, `logout` with Keycloak equivalents:

- `loginApi` тЖТ redirect to Keycloak via `signinRedirect()`
- `isTokenExpired` тЖТ check Keycloak user session
- `logout` тЖТ `signoutRedirect()`

### 2.5 Update `src/login/LoginPage.tsx`

On mount, check if this is a callback redirect from Keycloak (`signinRedirectCallback`). If so, process the callback and redirect to `/settlement`. Otherwise, redirect to Keycloak login.

### 2.6 Update `src/App.tsx`

Replace `isTokenExpired()` check with Keycloak session check. On load, initialize the Keycloak client and handle the callback flow.

1. **Backend startup**: Run `./mvnw clean package -DskipTests` and verify it compiles
2. **Keycloak connectivity**: Start the app and verify it fetches JWKS from Keycloak (`http://172.29.11.201:8086/realms/Invoicing%20System/protocol/openid-connect/certs`)
3. **Login flow**: Open `http://localhost:3000`, should redirect to Keycloak login page
4. **Token validation**: After Keycloak login, API calls should include the Keycloak JWT and backend should validate it
5. **Role enforcement**: Verify `@PreAuthorize("hasAnyRole('ADMIN', 'USER')")` still works with Keycloak roles
6. **Logout**: Verify logout redirects to Keycloak and clears the session

## Prerequisites (Keycloak Setup)

Ensure in Keycloak admin console:
1. Realm `Invoicing System` exists
2. Client `invoicingsystemclient` is configured as **public** with:
   - Valid Redirect URIs: `http://localhost:3000/*`
   - Web Origins: `http://localhost:3000`
   - Authentication flow: **Standard Flow** (Authorization Code) enabled
3. Users exist in the realm with roles `ADMIN` or `USER` assigned
